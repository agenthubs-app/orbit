Orbit Development Method

积木式模块化开发方法

Orbit 的开发边界不是按页面切，而是按能力切。每个能力像一块积木：先定契约，再用 mock 跑通产品路径，最后把 mock 换成真实业务组件。调用方不改，替换才算成功。

One Rule

页面不认识 mock，也不认识 provider

页面和 API route 依赖什么
service-factory.ts 与稳定的 contract.ts。
团队如何并行
一个模块一个边界。各团队在自己的 features/<module>/ 内完成 contract、service、mock、live 和测试。
什么时候可以替换 live
live 实现满足同一个 service interface，返回同一个 DTO，测试能在 mock、hybrid、live 模式下说明差异。

Why

它解决的问题

这个项目会有很多小能力：名片扫描、联系人导入、聊天摘要、活动推荐、跟进提醒、外部动作沙盒。它们成熟速度不同，也可能由不同人开发。如果页面直接 import 某个 mock，或者直接读某个 provider 的返回值，后面替换真实组件时就会牵动整条链路。

积木式方法的目标很朴素：让每个能力可以单独开发、单独测试、单独换掉。页面只看稳定输出，不关心背后是本地 fixture、OCR provider、邮箱 API，还是数据库。

Four Layers

一块积木的四层

1. Contract

定义 DTO、状态、错误码和输入输出。它是团队之间的协议，不写 provider 细节。

features/acquisition/.../contract.ts

2. Service Interface

定义调用方式。页面和 API route 只通过这个接口拿数据，不能绕过去读 fixture。

service.ts

3. Implementation

当前先放 deterministic mock。之后增加 live-service.ts、provider.ts、mappers.ts。

mock-service.ts / live-service.ts

4. Factory

根据 ORBIT_MODULE_MODE 或测试设置选择 mock、hybrid、live。缺 live 时必须显式失败。

service-factory.ts

App Page / API Route
  -> features/<module>/service-factory.ts
  -> service.ts interface
  -> contract.ts DTO
  -> mock-service.ts | live-service.ts

File Shape

推荐文件结构

features/acquisition/business-card-scan-ocr-mock/
  contract.ts          # DTO、状态、错误码
  service.ts           # service interface
  mock-service.ts      # 本地确定性实现
  live-service.ts      # 未来真实实现
  provider.ts          # OCR provider 适配，不被页面直接 import
  mappers.ts           # provider payload -> contract DTO
  validators.ts        # 输入、权限、字段校验
  LIVE_IMPLEMENTATION.md

features/acquisition/service-factory.ts
app/api/contact-drafts/business-card/scan/route.ts
app/(app)/app/contacts/new/page.tsx
tests/features/acquisition/...test.ts

命名可以随模块调整，但边界不能变：页面和 API route 通过 factory 拿 service；provider payload 必须先被 mapper 转成 contract DTO。

Workflow

一个模块的开发顺序

  1. 先写 contract。 写清楚输入、输出、状态、错误码、来源证据字段。字段不要贴 provider 原始结构。
  2. 再写 mock service。 mock 必须确定性、可重复、无外部副作用。名片扫描可以模拟 OCR，但不能真的调用 OCR。
  3. 注册 service factory。 默认 mock；live 未接入时返回 NOT_IMPLEMENTED，不要偷偷回退。
  4. 接 API route。 API 返回统一 envelope，错误走共享错误形状，不把异常对象直接丢给前端。
  5. 接产品页。 页面只使用 contract 字段。页面文案描述用户任务，不暴露 fixture、provider、内部 ID。
  6. 补测试。 测 contract、mock 确定性、API envelope、页面状态、无副作用证据。
  7. 再接 live。 live 只能替换 implementation，不应该要求页面和 API route 改调用方式。

Team Split

团队怎么分工

角色 主要改哪些文件 不要碰什么
模块开发者 features/<module>/、模块测试、对应 LIVE_IMPLEMENTATION.md 其他模块的 fixture、页面内部状态、harness 配置
页面开发者 app/(app)/app/**、页面组合测试、用户流程文案 mock-service 内部细节、provider 原始 payload
API 开发者 app/api/**/route.ts、API envelope 测试、错误映射 直接 import mock 或 live provider
live 接入者 live-service.ts、provider.ts、mappers.ts、validators.ts 改页面来适配 provider；这是边界失败的信号

Mock To Live

抽换真实组件的规则

可以变

provider、认证方式、外部 API、缓存、重试、限流、数据库、OCR 引擎。

不能变

service interface、contract DTO、页面读取字段、API envelope、无副作用默认行为。

mock-service.ts
  -> returns BusinessCardScanResult

live-service.ts
  -> calls OCR provider
  -> validates permission
  -> maps provider payload
  -> returns the same BusinessCardScanResult

如果接 live 时发现页面需要判断 providerName === "x"，说明 mapper 没有把 provider 差异收干净。修 mapper，不修页面。

Sprint Size

Sprint 粒度

一个 sprint 尽量只交付一个能力积木。比如“名片扫描 OCR mock”是一个 sprint，“名片字段人工复核”是另一个 sprint。不要把名片扫描、重复合并、外部联系人导入和联系人详情页塞在同一个 sprint 里。

合适

一个 contract、一个 service、一个 mock、一个 API route、一个页面入口或 dev 验证面。

偏大

一个 sprint 同时做多个 provider、多个页面和跨模块状态联动。

偏小

只改文案或只加 fixture，却没有可验证的接口边界。

Done

完成标准

  • contract 稳定。 字段命名能表达产品含义，不泄露 provider 原始字段。
  • mock 可重复。 同样输入得到同样结果，不访问网络、不写外部系统。
  • factory 是唯一入口。 页面、API route、聚合服务都不直接 import mock。
  • 失败显式。 live 未实现时返回 NOT_IMPLEMENTED，不假装成功。
  • 页面能说明用户任务。 用户看到的是“准备跟进 Kenji”，不是“fixture id 通过 harness 验证”。
  • 测试覆盖边界。 至少覆盖 service、API envelope、页面状态和 mock-to-live 替换约束。

Avoid

不要这样做

页面直接 import mock

这会让 mock 变成产品依赖，后面接 live 必然改页面。

contract 跟着 provider 摇摆

provider 返回什么就暴露什么，调用方会被外部系统绑住。

一个 sprint 做太多能力

冲突会增加，review 也会失真。最后没人知道到底是哪块积木坏了。

测试为了配置改代码

配置是用户意图。测试应该验证配置，而不是把配置改成测试喜欢的样子。

Example

名片扫描怎么落地

Business Card Scan OCR
  Contract:
    imageRef, extractedFields, confidence, sourceEvidence, reviewRequired

  Mock:
    local fixture -> deterministic extractedFields
    ocrProviderCalled: false
    externalNetworkRequested: false

  API:
    POST /api/contact-drafts/business-card/scan
    returns shared success/failure envelope

  Product:
    /app/contacts/new shows captured fields
    user confirms before contact draft enters review

  Live later:
    provider.ts calls OCR
    mappers.ts converts OCR payload
    service returns the same contract DTO
